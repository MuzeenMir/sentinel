## 1. Purpose

This plan describes how SENTINEL v2 (per [SRS-002](SRS-002.md) and
[SDD-002](SDD-002.md)) will be delivered: phased rollout, milestones, resource
allocation, risk register, quality gates, decommissioning, and the operating
rhythm during the revamp.

---

## 2. Operating principles for the revamp

These are non-negotiable for the duration of v2 work:

1. **Stop adding services.** Every new service requires an ADR and explicit
   approval. The revamp's primary axis is consolidation, not expansion.
2. **Honest scope in marketing.** Public-facing copy describes what is shipping,
   not what is planned. "AI-assisted Linux server hardening + intrusion
   detection" until v2 GA, then revisit.
3. **No work outside a phase.** If something is worth doing but not in any phase,
   it goes in the backlog (`docs/revamp/BACKLOG.md` — to be created), not into
   `main` directly.
4. **Phase 0 ships before Phase 1 starts.** Stabilization is a precondition;
   skipping it has been the failure mode in v1.
5. **Definition of Done is the same for every phase** (§7) — no exceptions for
   "we'll fix tests after."
6. **Two-person rule** for any change to: OPA bundles, model promotions, helm
   chart prod values, RLS policies, audit log schema.
7. **Daily 15-min standup, weekly 60-min review, monthly 90-min retro** — the
   only fixed meetings; everything else async.

## 5. Phase detail

### Phase 0 — Stabilize (4 weeks)

**Why first.** v1 audit shows the main branch breaks regularly (recent commits
are mostly damage control). Adding architecture on top of an unstable base
multiplies the chaos. Phase 0 is non-negotiable.

**Deliverables.**

- **CI gates as required checks** on every PR (lint, type-check, unit, smoke
  e2e under 15 min, gitleaks, Semgrep, Trivy, CodeQL).
- **Branch protection on `main`** (no direct push, ≥ 1 review, signed commits,
  status checks required) — implemented per [GIT-RESTRUCTURE](GIT-RESTRUCTURE.md).
- **Migrations idempotent** — recent commits show drift; freeze schema, write a
  migration smoke test that runs against a fresh DB on every PR.
- **Honest README** — strip "enterprise-grade AI security platform" language
  until v2 GA; replace with current shipping scope.
- **Secret hygiene sweep** — gitleaks history scan; rotate anything found;
  document outcome.
- **SBOM + cosign for current images** — even before consolidation starts.
- **OpenTelemetry boilerplate** wired into one service end-to-end as a template.
- **`docs/adr/` initialized** with ADR-000 (use ADRs); the in-flight v2 ADRs
  ratified.
- **`docs/revamp/BACKLOG.md` created** — single source of "later, not now."

**Entry criteria.** None — start immediately.

**Exit criteria.** §7 quality gates green for 7 consecutive days on `main`. No
unfinished migration in flight. README claims match shipping reality.

**Risks.**
- *Team morale: "this isn't progress."* Mitigation — phase is short and
  visible; daily updates show CI pass-rate climbing.

### Phase 1 — Consolidate (8 weeks)

**Goal.** Collapse the 11 v1 services into the 4 v2 services + LLM Gateway
shell, behind feature flags.

**Workstreams (parallel where possible).**

1. **`console` consolidation** (auth + api-gateway + frontend BFF) — week 1–4.
2. **`controller` consolidation** (alert-service + policy-orchestrator
   read-side + audit) — week 1–4.
3. **`analyzer` consolidation** (ai-engine + xai + stream skeleton) — week 2–5.
4. **`collector` consolidation** (data-collector + agent-grpc + skeletons for
   sensor adapters that come online in Phase 2) — week 2–5.
5. **`llm-gateway` shell** (no real model yet; honors interface, returns
   structured 503; audit + tenant-iso scaffolding) — week 3–6.
6. **Strangler-fig routing** via `USE_V2_*` feature flags per tenant — week 4–7.
7. **Internal canary tenant** running fully on v2 services — week 7–8.

**Entry criteria.** Phase 0 exit met.

**Exit criteria.** Internal canary tenant (`sentinel-internal`) runs entirely on
the four new services for 14 consecutive days. Zero P0/P1 regressions vs. v1.
v1 services still up and serving other tenants (decommission in Phase 2).

**Risks.**
- *Consolidation bugs (lost endpoints, missing auth rules)*. Mitigation —
  contract tests pinned to v1 OpenAPI; deny-list of endpoints intentionally
  removed, accept-list of everything else must keep working.
- *Tenant routing flag leaks data cross-tenant*. Mitigation — route flag is
  per-tenant in JWT issuance; analyzer + controller refuse to start without
  flag-aware tenancy lib.

---

### Phase 2 — Sensor swap (6 weeks)

**Goal.** Replace v1 in-house sensors with Falco, Suricata, Wazuh, OpenSCAP.
Decommission v1 sensor services.

**Workstreams.**

1. **Falco DaemonSet** + gRPC consumer + `falco_to_cim` mapping (weeks 1–3).
2. **Suricata sidecar/host install** + EVE consumer + `suricata_to_cim` (weeks
   1–3).
3. **Wazuh manager + agent integration** + API consumer + `wazuh_to_cim` (weeks
   2–4).
4. **OpenSCAP scheduler in agent** + ARF consumer + `openscap_to_cim` + finding
   storage (weeks 3–5).
5. **ATT&CK technique lookup bundles** for Falco rules and Suricata sigs
   (weeks 2–5).
6. **Ensemble retraining on CIM-shaped data** (weeks 3–6) — XGBoost + LSTM
   retrained; eval set extended.
7. **v1 XDP / data-collector / hids-agent removed** (weeks 5–6).

**Entry criteria.** Phase 1 internal canary stable.

**Exit criteria.** All four sensor pipelines feed CIM in production for ≥ 7
days. v1 sensor code moved to `archive/v1-sensors/`. Helm chart no longer
references v1 sensor services.

**Risks.**
- *Falco overhead unacceptable on customer hosts*. Mitigation — measured early
  on representative hosts; if too high, drop-in alternative is Tetragon (kept
  as ADR fallback).
- *Mapping bundles incomplete (low ATT&CK coverage)*. Mitigation — start with
  the top-50 most-frequent Falco rules / Suricata sigs covering the bulk of
  events; iterate.

---

### Phase 3 — LLM triage (6 weeks)

**Goal.** Production triage worker. Real Gemma 4 26B MoE behind LLM Gateway.
TurboQuant on KV cache.

**Workstreams.**

1. **vLLM cluster + Gemma 4 26B MoE deploy** (weeks 1–2). GPU node pool sized
   from the spike measurements.
2. **TurboQuant integration + measurement** (weeks 1–2). Compression ratio +
   accuracy delta logged.
3. **LLM Gateway full implementation** — routing, audit, prompt-injection scrub,
   tenant pools, schema validation, kill switch, budget enforcer (weeks 1–4).
4. **Triage worker in analyzer** — consumes `alerts.raw`, calls gateway,
   publishes `incidents.enriched` (weeks 2–4).
5. **Adversarial CI corpus** — initial 200 prompt-injection cases (weeks 2–4).
6. **Hallucination eval** — 500 incident ground-truth set (weeks 2–4); nightly
   run wired into CI.
7. **Console: AI-generated badge + provenance link** (weeks 4–5).
8. **Internal dogfood for 14 days** (weeks 5–6).

**Entry criteria.** Phase 2 sensor swap GA. Spike from §10 milestone M-S1
complete (triage cost/latency acceptable).

**Exit criteria.** 14 consecutive days of internal dogfood with: zero SAF-1
violations, ≥ 90% analyst acceptance of triage output, p95 triage latency
≤ 8 s, eval factuality ≥ 0.85.

**Risks.**
- *Hallucinated ATT&CK attribution causes wrong triage*. Mitigation — schema
  rejects unknown technique IDs; confidence gate routes uncertain to human;
  nightly eval blocks promotion on regression.
- *Cost per tenant uneconomic*. Mitigation — TurboQuant 5–6× KV win; per-
  tenant budget cap; degraded mode falls back to ML-score-only enriched
  incidents (gateway kill switch).

---

### Phase 4 — Edge LLM (6 weeks)

**Goal.** Gemma 4 e2b / e4b on agent profiles. Privacy-preserving on-device
explanation, NL log query, multimodal robotics.

**Workstreams.**

1. **`agent/inference/local_llm.py`** with llama.cpp + GGUF quantized models
   (weeks 1–3).
2. **Profile-gated activation** — server (e4b on), endpoint (e2b on), iot (off
   default), robotics (e4b multimodal), business_app (e2b) (weeks 2–3).
3. **Resource ceiling enforcer** — measure CPU/RAM/GPU; auto-disable per
   FR-AGT-7 priority order (weeks 2–4).
4. **Edge LLM update channel** separate from agent code (weeks 3–4).
5. **OpenSCAP finding explanation** rendered inline in console (weeks 4–5).
6. **NL log query** wired through agent → console (weeks 4–6).
7. **Battery / thermal field test** on real endpoint hardware (weeks 5–6).

**Entry criteria.** Phase 3 triage GA. Spike M-S2 (edge LLM viability) complete.

**Exit criteria.** Server + endpoint profiles ship with Edge LLM enabled by
default. Resource ceilings hold under load. No measurable user-perceptible
latency regression on a normal workstation.

**Risks.**
- *Edge LLM RAM burns battery on laptops*. Mitigation — endpoint profile
  default ON only when on AC power; falls back to server-side explanation
  otherwise.
- *Quantized model quality too poor on niche tasks*. Mitigation — eval set
  per-profile; if e2b fails, escalate to e4b or disable that feature.

---

### Phase 5 — Analyst Copilot (6 weeks)

**Goal.** RAG + tool-using copilot in the console. Read tools auto, write tools
human-clicked, every call audited.

**Workstreams.**

1. **`copilot/` module** in console with SSE chat, history, citations (weeks
   1–3).
2. **RAG retriever** over pgvector, RLS-bound (weeks 1–3).
3. **Tool registry split** (read auto / write button) (weeks 2–4).
4. **Copilot frontend page** with chat, tool result cards, write-tool buttons
   (weeks 2–5).
5. **Adversarial corpus extension** to copilot tool-use cases (weeks 3–5).
6. **Canary cohort opt-in** + 30-day measurement (weeks 5–6 + tail).

**Entry criteria.** Phase 3 triage GA (gateway and audit machinery in place).

**Exit criteria.** 30 days canary use. 100% of write-tool invocations preceded
by human click. Copilot tool-use adversarial corpus passes. No cross-tenant
exposure incidents.

**Risks.**
- *Copilot suggests confident-but-wrong actions*. Mitigation — tool surface
  split; suggestions are buttons, never auto-clicks.
- *Prompt injection escalates to tool misuse*. Mitigation — adversarial corpus
  in CI; gateway-side input wrapping; OPA-side action policy still applies.

---

### Phase 6 — Multimodal robotics + Compliance evidence (8 weeks)

**Goal.** Robotics profile uses Gemma 4 vision for scene anomaly. Compliance
engine becomes real via OpenSCAP + LLM-assisted control mapping.

**Workstreams.**

1. **Multimodal pipeline in agent** (camera frame capture + e4b vision + scene
   policy) (weeks 1–4).
2. **Scene anomaly use cases** — PPE compliance, unauthorized presence,
   equipment state (weeks 2–5).
3. **Compliance evidence pipeline** — OpenSCAP rule → control catalog mapping
   bundle (weeks 1–4).
4. **LLM-assisted evidence proposal** — Gemma reads raw configs/logs and
   proposes mappings; human accepts before counting as evidence (weeks 4–6).
5. **Audit-ready exports (PDF + JSON)** for SOC 2 + ISO 27001 + NIST CSF +
   PCI-DSS + GDPR (weeks 5–7).
6. **External compliance reviewer engagement** (weeks 7–8).

**Entry criteria.** Phase 4 (edge LLM) and Phase 5 (copilot) GA.

**Exit criteria.** Robotics profile demonstrably detects PPE violations and
unauthorized presence in a controlled environment. Compliance exports for SOC 2
and ISO 27001 reviewed by an external auditor and accepted as audit-ready.

**Risks.**
- *Multimodal false positives in robotics block legitimate operators*.
  Mitigation — runs in shadow for 30 days first; alerts only after 30-day
  precision/recall measurement on customer-tagged ground truth.
- *Compliance mappings wrong → audit failure*. Mitigation — every mapping
  reviewed by a human before counting; nothing auto-claims a control met.

## 6. Decommissioning plan (v1 services)

For each v1 service, the decommission gate is identical:

1. v2 equivalent has served 100% of tenants for ≥ 14 days, zero P0/P1
   regressions.
2. v1 service no longer referenced in any helm chart or compose file.
3. v1 endpoints (where API-facing) marked Sunset for one minor release window.
4. After window, v1 endpoints respond 410 Gone with a Sunset header pointing
   to the v2 equivalent.
5. v1 source code moved to `archive/v1-<area>/`, retained for reference.

| v1 service | Decommissioned in phase | Replaced by |
|---|---|---|
| api-gateway | Phase 1 | console |
| auth-service | Phase 1 | console |
| alert-service | Phase 1 | controller |
| dashboard-service | Phase 1 | console |
| ai-engine | Phase 1 (folded; same models) | analyzer |
| xai-service | Phase 1 (folded) | analyzer |
| policy-orchestrator | Phase 1 (folded; OPA added in Phase 1 too) | controller + OPA |
| data-collector | Phase 2 | collector + Falco/Suricata |
| xdp-collector | Phase 2 | collector + Falco |
| hids-agent | Phase 2 | collector + Wazuh |
| compliance-engine | Phase 6 | analyzer + OpenSCAP + LLM-assist |
| drl-engine (production path) | Phase 1 (write path removed) | research/drl/ (offline only) |
| stream-processing/flink-jobs | Phase 1 (skeleton) → Phase 2 (real) | analyzer Bytewax |

## 7. Quality gates (Definition of Done)

A change merges to `main` only when **all** of the following are green:

| Gate | Tooling |
|---|---|
| Lint | ruff (Python), eslint (TS), shellcheck (sh) |
| Format | ruff format, prettier |
| Type-check | mypy strict (Python), `tsc --noEmit` (TS) |
| Unit tests | pytest, vitest |
| Integration tests | pytest with testcontainers (Postgres, Redis, Kafka) |
| E2E smoke | playwright + docker-compose; < 15 min total |
| Security scans | gitleaks, Semgrep, Trivy (image), CodeQL |
| SBOM | generated and uploaded as artifact |
| Coverage | ≥ 75% statement per service touched (Phase 0 floor 70%; ratchet to 75% during Phase 1 `_lib/` consolidation) |
| AI-safety corpus (Phase 3+) | adversarial injections + cross-tenant + schema |
| Hallucination eval (Phase 3+) | nightly; promotion gate on regression > 5% |

A release ships only when the SRS-002 §10 acceptance criteria are met.

## 11. Tooling and process

- **Repo.** Per [GIT-RESTRUCTURE](GIT-RESTRUCTURE.md). Trunk-based with short-
  lived feature branches; conventional commits; release-please for changelog
  automation.
- **Issue tracking.** GitHub Issues with labels per phase (`phase/0-stabilize`
  through `phase/6-multimodal`). Project board per phase.
- **Design decisions.** ADRs in `docs/adr/` (markdown, sequentially numbered).
  Every PR that introduces a tech choice or architectural shift links its ADR.
- **PR template.** Required sections: summary, screenshots/recordings (UI),
  test evidence, ADR link if applicable, security note, rollback plan.
- **Code owners.** `CODEOWNERS` enforces review by domain expert (LLM gateway
  changes → AI lead; OPA bundles → Security; agent → endpoint owner).
- **Release cadence.** Calendar-versioned (`vYYYY.MM.MICRO`) on a 2-week
  cadence during phases; semantic-versioned tags for SDK/agent (`vMAJOR.MINOR.PATCH`).

## 12. Maintenance and on-call (during the revamp)

- v1 still serves customers throughout Phases 0–2. v1 stays patched for
  security and P0 bugs only; no new v1 features.
- v2 internal canary tenant from Phase 1 onward — that's where dogfood
  pressure surfaces issues.
- On-call rotation: weekly, primary + secondary; runbooks for each consolidated
  service in `docs/operations/runbooks/`.
- Incident response per [v1 incident process] until Phase 6, then revisit.

## 14. v2 GA definition

SENTINEL v2.0.0 is GA when:

1. SRS-002 §10 acceptance criteria all met.
2. Phases 0–5 completed; Phase 6 has shipped robotics scene anomaly *or* the
   compliance evidence exports (one-of-two acceptable for v2.0; the other for
   v2.1).
3. External red team has completed at least two engagements with no
   unresolved Critical or High findings.
4. At least one external customer has run on v2 for ≥ 30 days with no SAF-1
   violations and no data-isolation incidents.
5. Documentation set (revamp + operations + API + security) reviewed and
   stamped by a non-author engineer.
6. SBOM published; container images cosigned with SLSA L3 provenance.
