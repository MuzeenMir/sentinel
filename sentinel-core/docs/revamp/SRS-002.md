## 1. Introduction

### 1.1 Purpose

This document is the authoritative requirements baseline for the SENTINEL **v2
revamp**. It defines what the system must do after the consolidation, sensor swap,
and AI-augmented SOC capabilities described in [SDP-002](SDP-002.md) are complete.
All design (SDD-002), implementation, and testing artifacts trace to requirements
in this document.

### 1.2 Scope (revised)

SENTINEL v2 is an **AI-assisted Linux server and endpoint detection-and-response
platform** with an optional cloud control plane. It is **not** a SIEM, **not** an
EDR replacement, **not** a generic "enterprise security platform." Concrete scope:

**In scope (v2):**
- Detection-and-response on Linux servers (Ubuntu 22.04+ / RHEL 9+ / Debian 12+).
- Detection-and-response on Linux endpoints (developer workstations, kiosks).
- Detection-and-response on selected IoT/robotics gateways (Jetson Orin, x86
  industrial PCs).
- AI-assisted alert triage, narrative correlation, and analyst copilot powered by
  Gemma 4 + TurboQuant.
- On-device LLM explanation and NL queries on capable agents.
- CIS-benchmark hardening recommendations and tracked remediation.
- Compliance evidence-mapping for SOC 2, ISO 27001, NIST CSF, PCI-DSS, GDPR
  (advisory; not a substitute for a GRC platform).
- Multi-tenant cloud control plane (managed and self-hosted variants).

**Explicitly out of scope (v2):**
- Windows endpoints (deferred to v3).
- macOS endpoints (deferred to v3).
- Network-device management (switches, routers).
- Replacement of upstream sensors — Falco, Suricata, Wazuh, OpenSCAP are
  *integrated*, not reimplemented.
- Autonomous DRL-driven policy enforcement — research track only, not a v2 product
  feature (see §3.10 Deprecations).
- In-house eBPF kernel programs — replaced by Falco/Tetragon in §3.2.
- Custom Flink cluster — replaced by lightweight Python stream consumers in §3.3.

## 2. Overall description (revised)

### 2.1 Product perspective

SENTINEL v2 is a thin AI-augmented control plane on top of best-of-breed open-source
sensors. It does **not** own the sensor layer. The system has three deployment
shapes:

1. **Self-hosted single-tenant.** All services + sensors on customer infrastructure.
   Air-gappable for regulated tenants.
2. **Managed multi-tenant cloud.** Control plane runs in SENTINEL's AWS account;
   customer agents call out via mTLS.
3. **Hybrid.** Agents and sensors on customer infra; control plane managed by
   SENTINEL. Per-tenant LLM routing decides whether inference happens on customer
   GPU, SENTINEL GPU, or on-device.

### 2.2 Product functions (v2)

After consolidation, SENTINEL v2 performs:

1. **Sensor integration** — ingests events from Falco (host syscalls), Suricata
   (network), Wazuh (FIM/HIDS), OpenSCAP (compliance scans), and the SENTINEL agent
   (lightweight metadata + on-device LLM output).
2. **Stream normalization** — events from all sensors normalized into a Common
   Information Model and published to per-tenant Kafka topics.
3. **Threat detection** — supervised XGBoost + LSTM ensemble (retained from v1,
   trained on the normalized CIM); unsupervised Isolation Forest + autoencoder for
   drift/novelty.
4. **AI triage** — Gemma 4 26B MoE deduplicates, correlates, ATT&CK-maps, and
   narrates raw alerts into incidents with strict structured output. Advisory only.
5. **Analyst copilot** — Gemma 4 31B dense + RAG over per-tenant alert/incident
   history + read-only tool calls into SENTINEL APIs. Write tools require human
   approval.
6. **Edge LLM** — Gemma 4 e2b/e4b on capable agents for on-device explanation, NL
   log query, multimodal scene anomaly (robotics).
7. **Policy decision** — OPA evaluates declarative policy + ML scores + human
   approval state to choose enforcement action.
8. **Policy enforcement** — applies firewall rules via vendor adapters (iptables,
   nftables, AWS SG, Azure NSG, GCP Firewall, Cloudflare).
9. **Hardening** — runs OpenSCAP-driven CIS scans; tracks remediation; explains
   each finding via on-device LLM.
10. **Compliance** — maps OpenSCAP evidence + alert history to control catalogs
    (SOC 2, ISO 27001, NIST CSF, PCI-DSS, GDPR); generates audit-ready exports.
11. **Console** — React SPA with dashboards, incident triage, analyst copilot pane,
    policy editor, hardening posture, compliance reports, audit log viewer.

### 2.3 User classes (additions to v1)

The v1 user classes (Tenant Admin, Security Analyst, Compliance Officer, Auditor,
End User) remain. v2 adds:

| Role | Description | Access level |
|---|---|---|
| **AI Reviewer** | Calibrates and red-teams LLM behavior (triage quality, copilot tool use). Reviews flagged hallucinations. | Read+ on incidents; write on LLM eval datasets and guardrail config |
| **Policy Author** | Writes OPA policy bundles; approves policy promotions through staging→production | Write on policy bundles; promote requires two-person rule |

### 2.4 Operating environment (revised)

| Layer | v1 baseline | v2 requirement |
|---|---|---|
| Container runtime | Docker | Docker **and** Kubernetes 1.29+ (production deploys Kubernetes) |
| Database | PostgreSQL 14 | PostgreSQL 16 with `pgvector` extension and Row-Level Security on every tenant-scoped table |
| Streaming | Kafka 3.x + (claimed) Flink | Kafka 3.x **only**; stream processing in Python consumers ([Bytewax](https://bytewax.io)) |
| Cache | Redis 7 | Redis 7 (per-tenant key prefixes enforced by middleware, validated in CI) |
| Observability | ad-hoc logs | OpenTelemetry traces + metrics + logs; OTLP-compatible backend (Tempo/Loki/Mimir or vendor) |
| LLM serving | n/a | vLLM with TurboQuant for hosted models; llama.cpp with GGUF quantization for edge |
| Sensors | in-house | Falco ≥ 0.41, Suricata ≥ 8.0, Wazuh ≥ 5.0, OpenSCAP ≥ 1.4 |
| Policy decision | bespoke | Open Policy Agent ≥ 0.70 with bundled OPA policies |

### 2.5 Constraints

- **Air-gap support is mandatory.** Every required artifact (model weights,
  sensor binaries, OPA bundles, container images) must be installable from a
  signed offline bundle.
- **No customer telemetry leaves the customer's data plane** unless the customer
  opts in to managed-cloud routing.
- **All LLM inference is observable and auditable.** Prompt + response are hashed;
  full content is retained per-tenant retention policy (default 30 days).
- **No autonomous write actions from any AI component.** Every state-changing tool
  call requires explicit human confirmation, recorded in the audit log.

### 2.6 Assumptions and dependencies

- Customer hosts can run a single privileged container (Falco) or kernel module.
  Hosts where this is forbidden are out of scope for host-syscall detection.
- Customer's Kubernetes cluster has GPU nodes available for hosted LLM inference,
  OR customer accepts managed-cloud LLM routing, OR customer disables LLM features.
- Gemma 4 weights are downloadable under their open-weights license at the time of
  deploy; SENTINEL re-distributes weights inside its signed offline bundle for
  air-gap installs (subject to license review).

## 3. Functional requirements

Each requirement carries a stable ID `FR-<service>-<n>`. Priority: **MUST**, **SHOULD**, **MAY**.
Numbering restarts per service.

### 3.1 collector (consolidated: ingestion + agent management)

**FR-COL-1 (MUST).** Accept events from Falco via gRPC/Unix socket using Falco's
native output channel. No re-implementation of syscall capture.

**FR-COL-2 (MUST).** Accept events from Suricata via EVE JSON (file tail or
Redis output). Support both inline and IDS-mode deployments.

**FR-COL-3 (MUST).** Accept events from Wazuh via the Wazuh API (events endpoint)
or by tailing Wazuh's archives.json.

**FR-COL-4 (MUST).** Accept events from OpenSCAP scan results in ARF XML format.

**FR-COL-5 (MUST).** Accept events from the SENTINEL agent over mTLS (HTTP/2 +
protobuf). Agent events include host metadata, process snapshots, optional
on-device LLM outputs, and profile-specific events (USB, FIM beyond Wazuh, scene
anomaly for robotics).

**FR-COL-6 (MUST).** Normalize all sensor events into a Common Information Model
(CIM) with a stable schema versioned in `proto/cim/v2/`. CIM fields: tenant_id,
host_id, timestamp_ns, source (falco|suricata|wazuh|openscap|sentinel-agent),
event_type, raw_payload, normalized_fields, attack_techniques (initial guess from
sensor metadata).

**FR-COL-7 (MUST).** Tag every event with `tenant_id` from the agent's mTLS client
certificate; reject events whose payload claims a different tenant_id.

**FR-COL-8 (MUST).** Publish CIM events to Kafka with topic
`<env>.tenant.<tenant_id>.events.<source>` (per-tenant topics — no global topics
for tenant-scoped data).

**FR-COL-9 (SHOULD).** Backpressure when downstream Kafka is unavailable: buffer
locally up to a configurable disk quota, drop with structured warning when
exhausted, surface as a tenant-visible health event.

**FR-COL-10 (MUST).** Expose `/healthz` and `/readyz` distinct from each other.
`/readyz` returns 503 if any required sensor connection is dead.

**FR-COL-11 (MUST).** Issue, rotate, and revoke per-agent client certificates via
an internal CA (cert-manager-compatible). Default rotation: 90 days.

### 3.2 analyzer (consolidated: feature extraction + ML detection + XAI + triage)

**FR-ANA-1 (MUST).** Consume CIM events from `tenant.<id>.events.*` topics and
extract features into a feature store (Redis for hot windows, PostgreSQL for
historical).

**FR-ANA-2 (MUST).** Score every event (or every k-second window) with the
ensemble (XGBoost + LSTM + Isolation Forest + autoencoder, retained from v1 with
retraining on CIM data).

**FR-ANA-3 (MUST).** Refuse to start if any required model file is missing or its
SHA-256 does not match the expected manifest. **No silent fallback to untrained
defaults** (closes a v1 risk).

**FR-ANA-4 (MUST).** For every detection, produce a SHAP-based explanation
(features ranked by contribution) and persist alongside the alert.

**FR-ANA-5 (MUST).** Publish raw alerts to `tenant.<id>.alerts.raw`. The alert
schema includes ensemble scores, SHAP top-10, sensor source, related event IDs,
and any sensor-supplied ATT&CK technique IDs.

**FR-ANA-6 (MUST).** Run a **triage worker** that consumes `alerts.raw` and calls
the LLM Gateway with model `triage-v1` (Gemma 4 26B MoE). The worker enforces
strict JSON output via function calling and produces an enriched incident with:
{incident_id, narrative, attack_techniques[], confidence, deduplicated_event_ids[],
suggested_response, requires_human_review}.

**FR-ANA-7 (MUST).** Publish enriched incidents to `tenant.<id>.incidents.enriched`.

**FR-ANA-8 (MUST).** Drop or escalate to human review any LLM output where: (a)
JSON schema validation fails, (b) `attack_techniques[]` contains unknown IDs, (c)
`confidence` < configurable threshold (default 0.65).

**FR-ANA-9 (MUST).** Tag every triage call with `prompt_hash`, `output_hash`,
`model_version`, `latency_ms` for the audit log.

**FR-ANA-10 (SHOULD).** Cluster duplicate incidents within a configurable window
(default 10 minutes) before publishing — never duplicate-paginate the same
incident to the analyst.

### 3.3 controller (consolidated: alerts + policy decision + policy enforcement + DRL research interface)

**FR-CTR-1 (MUST).** Consume `incidents.enriched`, persist to PostgreSQL, run the
alert lifecycle state machine (new → acknowledged → investigating → resolved /
false-positive / escalated).

**FR-CTR-2 (MUST).** Dispatch notifications via channels configured per-tenant
(email, Slack, Teams, webhook, PagerDuty). Per-channel rate limits.

**FR-CTR-3 (MUST).** Use OPA as the policy decision point. For every incident
that *could* trigger enforcement, controller queries OPA with
{incident, tenant_policy_bundle, env, ml_scores, llm_suggestion} → OPA returns
{action, justification, requires_human_approval}.

**FR-CTR-4 (MUST).** Apply policy actions through firewall adapters (iptables,
nftables, AWS SG, Azure NSG, GCP Firewall, Cloudflare). Each adapter implements a
common interface: `apply(rule) -> rule_id`, `revoke(rule_id)`, `list() -> [rule]`.

**FR-CTR-5 (MUST).** Every applied rule has: `tenant_id`, `incident_id`,
`approver_user_id` (or `auto`), `expires_at` (default 24h, no auto-permanent
rules), `revert_token`. Provide one-click revert in the console.

**FR-CTR-6 (MUST).** When OPA returns `requires_human_approval`, hold the action
in `pending_approval` state, notify on-call, and time out after a configurable
window (default 4h) with auto-revert on timeout.

**FR-CTR-7 (MUST).** **No automatic write actions originate from LLM output.**
LLM-suggested response in the enriched incident is presented to the analyst as a
suggestion; OPA may permit auto-action only when ML score > threshold and the
suggestion has a sensor-grounded indicator (not LLM-only attribution).

**FR-CTR-8 (MUST).** Maintain an immutable audit log of every policy action and
every approval (PostgreSQL `audit_log` table with append-only constraint and
periodic external-storage export).

**FR-CTR-9 (SHOULD).** Expose a **DRL research interface** that records
state/action/reward tuples from controller decisions for offline RL training. The
PPO agent from v1 retains read-only access; **no write path back into the
production controller** until DRL graduates from research (see §7 Deprecations).

### 3.4 console (consolidated: API gateway + auth + frontend BFF + admin UI)

**FR-CON-1 (MUST).** Single edge service handling auth (JWT + OIDC + SAML +
TOTP/MFA + SCIM), API routing, rate limiting, and serving the React SPA bundle.
Replaces v1 separate api-gateway + auth-service.

**FR-CON-2 (MUST).** All API responses are tenant-scoped via JWT `tenant_id` claim
*and* PostgreSQL Row-Level Security. RLS is the security boundary; application
checks are defense-in-depth.

**FR-CON-3 (MUST).** Provide an **Analyst Copilot** pane: chat UI backed by Gemma
4 31B dense + RAG over the tenant's incident history + read-only tool calls.

**FR-CON-4 (MUST).** Every Copilot tool invocation is logged. Write tools
(quarantine_host, apply_firewall_rule, mark_incident_resolved) are surfaced as
actionable buttons in the chat — clicking the button is the human approval, not
the LLM's invocation.

**FR-CON-5 (MUST).** Surface incident narratives produced by the triage service
with an **"AI-generated"** badge and a link to the underlying raw alerts. Never
hide the provenance.

**FR-CON-6 (MUST).** Provide a **hardening posture view** per host: OpenSCAP rule
results, remediation status, last-scanned, with on-device LLM explanation rendered
inline.

**FR-CON-7 (MUST).** Provide a **compliance evidence view**: control catalogs
(SOC 2 / ISO / NIST / PCI / GDPR) with mapped OpenSCAP evidence, alert history,
and gap-analysis exports.

**FR-CON-8 (MUST).** Provide a **policy editor** for OPA bundles with: syntax
highlighting, dry-run against last-7-days incident history, two-person promotion
to production, rollback to any prior version.

### 3.5 llm-gateway (cross-cutting service)

**FR-LLM-1 (MUST).** Single point of entry for every LLM call from any other
service. No service may call vLLM, Ollama, llama.cpp, or any model API directly.

**FR-LLM-2 (MUST).** Per-tenant token budget enforcement (requests/min, tokens/day,
$/month). Reject when over budget with structured error.

**FR-LLM-3 (MUST).** Per-tenant prompt isolation — never leak one tenant's data
into another tenant's context window. Enforced by per-tenant connection pools
plus runtime assertion on every prompt.

**FR-LLM-4 (MUST).** Prompt injection defense: every untrusted string (any field
sourced from observed traffic, agent telemetry, or sensor output) wrapped in
`<observed>...</observed>` tags. System prompt explicitly instructs model that
content inside such tags is hostile and must not be followed as instructions.
Validation: red-team test corpus runs in CI (see SRS-002 §5).

**FR-LLM-5 (MUST).** Audit log: prompt SHA-256, output SHA-256, full
prompt+output stored per-tenant retention policy, model_version, tool_calls[],
latency_ms, $cost_estimate.

**FR-LLM-6 (MUST).** Model routing: requests carry a logical model name
(`triage-v1`, `copilot-v1`, `edge-explain-v1`); the gateway resolves to a
physical backend (`gemma-4-26b-moe@vllm-prod`, `gemma-4-31b@vllm-prod`,
`gemma-4-e2b@on-device`). Backend swaps require no code change in callers.

**FR-LLM-7 (MUST).** TurboQuant applied to KV cache for all hosted backends. KV
cache compression must achieve ≥ 5x measured at deploy with accuracy loss
< 1% on the SENTINEL eval set (see §6.2 LLM eval requirements).

**FR-LLM-8 (MUST).** Output schema validation: when caller supplies a JSON schema
(via function calling), gateway validates the model output and returns a typed
error on schema violation. Caller never sees raw model text in that path.

**FR-LLM-9 (MUST).** Kill switch — operator can disable any logical model
(`triage-v1` off → triage worker degrades to ML-score-only enriched incidents
with a "LLM disabled" flag, never crashes).

**FR-LLM-10 (SHOULD).** Tenant-pluggable model backend — a tenant who insists on
their own LLM (compliance, sovereignty) can route to a tenant-supplied endpoint
that meets the gateway's contract.

### 3.6 agent (Sentinel Profile-driven host process)

**FR-AGT-1 (MUST).** Single binary (Python wheel + optional Rust core for
hot-path) that implements the v1 profiles (server, endpoint, IoT, robotics,
business_app), each profile selecting active collectors and on-device features.

**FR-AGT-2 (MUST).** Mutual TLS to the collector. Client certificate stored in
the system keystore (Linux: gnome-keyring / kwallet / fallback to encrypted file
with TPM-bound passphrase where available).

**FR-AGT-3 (MUST).** Local event buffer with disk quota and FIFO drop policy.

**FR-AGT-4 (MUST).** Profile-gated **Edge LLM**:
- `endpoint`: Gemma 4 e2b enabled by default; explanations stay on-device.
- `server`: Gemma 4 e4b enabled by default; NL log query and FIM rationale.
- `iot`: Edge LLM disabled by default; opt-in flag for capable gateways.
- `robotics`: Gemma 4 e4b multimodal; scene anomaly (PPE, unauthorized presence,
  equipment state).
- `business_app`: Gemma 4 e2b for NL audit log query.

**FR-AGT-5 (MUST).** Edge LLM outputs that the agent ships upstream are clearly
marked `agent_llm_inference` in the CIM event; upstream services treat them as
**suggestions**, never as ground truth.

**FR-AGT-6 (MUST).** Auto-update via signed bundle (cosign verification, SLSA L3
provenance). Operator can pin version, disable auto-update, and roll back.

**FR-AGT-7 (MUST).** Resource ceiling enforced (configurable, defaults: ≤ 5%
sustained CPU on a 4-core host, ≤ 500 MB RSS without Edge LLM, ≤ 2 GB with e2b,
≤ 4 GB with e4b). Exceeding ceiling triggers structured warning and feature
auto-disable in priority order: multimodal → LLM → optional sensors → core.

### 3.7 hardening (capability provided by analyzer + agent + console)

**FR-HRD-1 (MUST).** Schedule and run OpenSCAP scans against per-profile CIS
benchmarks; persist results.

**FR-HRD-2 (MUST).** For each finding, the agent's Edge LLM produces a one-
paragraph explanation; the explanation is stored with the finding.

**FR-HRD-3 (MUST).** Generate remediation as either: (a) Ansible playbook,
(b) shell script, (c) configuration manager-specific (Puppet/Chef on roadmap).
Remediation requires explicit human approval; never auto-applied.

**FR-HRD-4 (MUST).** Track remediation state per finding: open → planned →
applied → verified → closed.

### 3.8 compliance (capability provided by analyzer + console)

**FR-CMP-1 (MUST).** Map OpenSCAP rule results, alert history, and audit log
events to control catalogs for SOC 2, ISO 27001 Annex A, NIST CSF, PCI-DSS,
GDPR. Mapping bundles versioned and reviewed.

**FR-CMP-2 (MUST).** Generate audit-ready exports (PDF + machine-readable JSON)
per control with linked evidence.

**FR-CMP-3 (MUST).** Mark gaps clearly; never claim a control is met without
linkable evidence.

**FR-CMP-4 (SHOULD).** LLM-assisted evidence gathering: Gemma 4 reads raw
configs/logs and proposes mappings to controls. All proposals require human
acceptance before counting as evidence.

### 3.9 observability (cross-cutting)

**FR-OBS-1 (MUST).** Every service emits OpenTelemetry traces with W3C trace
context propagation across HTTP, gRPC, and Kafka headers.

**FR-OBS-2 (MUST).** A single trace must follow: agent event → collector → kafka
→ analyzer (feature + score + triage) → controller → enforcement → notification.
This is acceptance-tested.

**FR-OBS-3 (MUST).** Standard metrics emitted: request rate, error rate, latency
p50/p95/p99 per endpoint; per-tenant LLM token spend; per-tenant alert volume;
ensemble model latency; sensor liveness.

**FR-OBS-4 (MUST).** Structured logging (JSON) with `tenant_id`, `trace_id`,
`incident_id` fields where applicable. No PII in logs at INFO level.

### 3.10 deprecations (what v2 removes)

**FR-DEP-1 (MUST).** v1 in-house XDP/eBPF collector code is **removed** from the
shipping product (retained in `archive/v1-xdp/` for reference). Replaced by Falco.

**FR-DEP-2 (MUST).** v1 PyFlink jobs are **removed** from production deploys.
Stream processing in Bytewax (Python) consumers within the analyzer service.

**FR-DEP-3 (MUST).** v1 PPO/DRL agent is **removed from the production decision
path**. Retained behind a feature flag in `research/drl/` for offline experiments
only. SDD-002 §10 documents the kill switch.

**FR-DEP-4 (MUST).** v1 separate api-gateway and auth-service are **merged** into
the console service. Endpoints retain their paths for backward compatibility for
one minor release after Phase 1 GA.

**FR-DEP-5 (MUST).** v1 separate alert-service is **merged** into the controller
service. Same backward-compatibility window.

**FR-DEP-6 (MUST).** v1 in-house compliance engine (260 LOC of effective
checkbox-stubs) is **removed**. Replaced by OpenSCAP-driven evidence collection
plus the LLM-assisted mapping in §3.8.

## 4. Non-functional requirements (raised baselines)

### 4.1 Performance

| ID | Requirement | Target |
|---|---|---|
| NFR-PERF-1 | Sensor-event ingest throughput per collector pod | ≥ 50,000 events/sec sustained |
| NFR-PERF-2 | Detection end-to-end latency (sensor event → ML score persisted) p95 | ≤ 2 seconds |
| NFR-PERF-3 | Triage latency (alert.raw → incidents.enriched) p95 | ≤ 8 seconds |
| NFR-PERF-4 | Analyst Copilot first-token latency p95 | ≤ 1.5 seconds |
| NFR-PERF-5 | Edge LLM (e2b) explanation latency p95 on 8GB Jetson Orin Nano | ≤ 3 seconds |
| NFR-PERF-6 | Console SPA Time-to-Interactive on a 100-incident page | ≤ 2 seconds |
| NFR-PERF-7 | Agent CPU on idle host | ≤ 1% sustained |
| NFR-PERF-8 | OPA decision latency p95 | ≤ 50 ms |

### 4.2 Reliability

| ID | Requirement | Target |
|---|---|---|
| NFR-REL-1 | Control plane availability (managed cloud) | 99.9% / quarter |
| NFR-REL-2 | Single-node agent crash recovery | ≤ 30 seconds, no event loss within local buffer |

## 5. AI safety requirements (new in v2)

These requirements implement the NIST AI RMF and OWASP LLM Top 10 baselines for
SENTINEL's LLM-using components.

**SAF-1 (MUST).** **No autonomous write action from LLM output.** Every state-
changing tool call requires explicit human approval recorded in the audit log.

**SAF-2 (MUST).** **Provenance always visible.** Any LLM-generated content shown
in the console carries an "AI-generated" badge with one click to the underlying
raw evidence.

**SAF-3 (MUST).** **Confidence-gated escalation.** Triage outputs with confidence
< threshold are routed to human review and are not used to update detection
priors.

**SAF-4 (MUST).** **Adversarial CI corpus.** A versioned corpus of prompt-
injection attempts (initially 200 cases, growing) runs against the triage worker
and analyst copilot in CI. Pass criteria: zero cases successfully induce out-of-
schema output, schema-violating tool calls, or cross-tenant data exposure.

**SAF-5 (MUST).** **Hallucination evaluation.** A versioned eval set of
incident-narrative ground truth (initially 500 incidents) runs nightly. Metrics:
ATT&CK technique precision/recall, narrative factuality (LLM-judge with human
spot-check), suggested_response appropriateness. Regression > 5% blocks model
promotion.

**SAF-6 (MUST).** **Per-tenant prompt isolation test.** A CI test attempts to
elicit tenant-A's data while authenticated as tenant-B; must fail closed at the
LLM Gateway.

**SAF-7 (MUST).** **Model card for every deployed model.** Documented in
`docs/models/`. Updated on every model swap.

**SAF-8 (MUST).** **Kill switch tested in CI.** Disabling `triage-v1` must result
in degraded-but-functional incidents (ML score only) without any 5xx errors.

**SAF-9 (MUST).** **Red-team cadence.** External red-team review of LLM
components every 6 months; findings tracked in [v1
MODEL-RED-TEAM](../MODEL-RED-TEAM.md) and SDD-002 §10.

**SAF-10 (SHOULD).** **Watermarking / provenance** of any LLM-authored artifact
that leaves the system (export reports, suggested playbooks). Format TBD; track
in ADR.

## 6. LLM operational requirements (new in v2)

**OPS-1 (MUST).** **Per-tenant LLM cost cap.** Daily token spend cap configurable
per tenant; soft alert at 80%, hard cutoff at 100% with degraded mode.

**OPS-2 (MUST).** **Model versioning.** Every model has a stable logical name
(e.g., `triage-v1`) and a versioned physical name (e.g., `gemma-4-26b-moe-2026.04`).
Promotion is two-person; rollback is one-click.

**OPS-3 (MUST).** **Eval gate before promotion.** A new model version must pass
the SAF-5 hallucination eval and the SAF-4 adversarial corpus before becoming
the default for any tenant.

**OPS-4 (MUST).** **Canary rollout.** New model versions roll out to a tenant
opt-in canary cohort first; metric monitoring window ≥ 7 days before general
availability.

**OPS-5 (MUST).** **TurboQuant validation** at deploy: KV cache compression ratio
measured and logged; SAF-5 eval re-run on the quantized model; drop in factuality
> 1% blocks deploy.

**OPS-6 (MUST).** **Edge LLM update channel.** Agents pull new edge model bundles
on a separate, slower channel from agent code; rollback independent.

**OPS-7 (MUST).** **Hosted LLM observability.** GPU utilization, memory pressure,
KV-cache hit rate, request queue depth, per-tenant tokens/sec all visible in the
operator dashboard.

**OPS-8 (SHOULD).** **Cost attribution.** Every LLM call tagged with
`{tenant_id, service, logical_model, purpose}` so per-tenant per-feature spend
is measurable.

## 10. Acceptance criteria (release-level)

A v2 release is acceptable when:

1. All MUST requirements above pass their referenced tests in CI on `main`.
2. The v1 services listed in FR-DEP-1..6 are absent from the production manifest.
3. The triage service has run for ≥ 14 days on internal dogfood with zero SAF-1
   violations and ≥ 90% triage acceptance rate by analysts.
4. The adversarial corpus (SAF-4) has zero successful injections.
5. The hallucination eval (SAF-5) shows ≥ 0.85 ATT&CK technique F1 and ≥ 0.80
   factuality on the held-out set.
6. End-to-end trace (FR-OBS-2) is verifiable in the observability backend on a
   test incident.
7. Helm chart deploys clean to a fresh k8s cluster (NFR-PORT-1) and the smoke
   suite passes.
8. SBOM (NFR-SEC-6) and signed images (NFR-SEC-3) attached to the release.
