# ADR-000 — Architecture Decision Record process

- **Status:** Accepted
- **Date:** 2026-04-19
- **Deciders:** SENTINEL tech lead, CODEOWNERS of `/sentinel-core/docs/adr/`
- **Supersedes:** None
- **Superseded by:** None

## Context

SENTINEL v1 accumulated architecture decisions in commit messages, Slack threads, and private notes. The April 2026 audit (`CODE-REVIEW-main-2026-04-18.md`) found that service boundaries, ML model choices, and deployment targets had drifted from any written record. The v2 revamp (`docs/revamp/`) re-grounds the system, and it needs a lightweight but durable decision log to avoid repeating the drift.

This ADR defines how ADRs are written, reviewed, and ratified in this repository.

## Decision

1. **Location.** ADRs live in `docs/adr/ADR-NNN-kebab-title.md`. Pre-flatten, that resolves to `sentinel-core/docs/adr/`. After the Phase 0 flatten lands, the files move to `docs/adr/` at the repo root and `sentinel-core/docs/adr/` becomes a symlink or redirect.

2. **Numbering.** Sequential zero-padded integers (`ADR-000`, `ADR-001`, …). Never reuse a number. If an ADR is withdrawn before acceptance, mark it **Withdrawn** and keep the number.

3. **Template.** Every ADR uses the following frontmatter block and sections. Missing sections fail CI (checked by a lint hook to be added in Phase 0).

   ```markdown
   # ADR-NNN — <title>

   - **Status:** Proposed | Accepted | Deprecated | Superseded | Withdrawn
   - **Date:** YYYY-MM-DD
   - **Deciders:** <names or roles>
   - **Supersedes:** ADR-XXX | None
   - **Superseded by:** ADR-YYY | None

   ## Context
   ## Decision
   ## Consequences
   ## Alternatives considered
   ## References
   ```

4. **Lifecycle.**
   - **Proposed** — open as a PR. Discussion happens in the PR, not the file.
   - **Accepted** — merged to `main` after review rule below. Status line updated in the same PR that merges it.
   - **Deprecated** — still historically relevant but no longer recommended. Keep the file.
   - **Superseded** — replaced by a newer ADR. Fill in the **Superseded by** field; the replacement's **Supersedes** field points back. Keep the file.
   - **Withdrawn** — proposed then abandoned without acceptance. Keep the file for institutional memory.

5. **Ratification rule.** To move from *Proposed* to *Accepted*:
   - Minimum **one CODEOWNERS approval** from `/sentinel-core/docs/adr/` (and `/docs/adr/` post-flatten).
   - **Two-person rule** applies if the ADR affects: OPA policy surface, model promotion, Helm production values, Postgres RLS, or audit schema. In those cases a second approval from the relevant CODEOWNERS (security or infra) is mandatory.
   - No open unresolved conversations on the PR.
   - Any affected specification in `docs/revamp/` (SRS, SDD, SDP) is updated in the same PR.

6. **Scope.** ADRs record *architecture* decisions, not implementation detail. Good ADR candidates: framework choice, data-plane topology, security boundary, failure-domain split, supply-chain policy. Bad ADR candidates: variable naming, single-function refactors, transient bug fixes.

7. **Immutability of Accepted ADRs.** Once Accepted, an ADR's **Decision** and **Context** sections are not edited except for typos. New information lives in a new ADR that either supersedes or references the old one. **Consequences** may be amended with a dated addendum.

8. **Discoverability.** A one-line index at `docs/adr/README.md` lists every ADR with its status and title. The index is regenerated by a script (`scripts/gen_adr_index.sh`, Phase 0 deliverable).

## Consequences

- Positive: every non-trivial architectural choice leaves a written trail tied to a PR, a date, and a reviewer set.
- Positive: Conventional Commits scope `adr` routes commits cleanly to release notes.
- Negative: small overhead for every cross-cutting change. We accept this — the v1 audit showed the cost of *not* writing things down is higher.
- Migration cost: in-flight v2 ADRs (ADR-001 through ADR-010, referenced in `SDD-002.md`) must be ratified in Phase 0.

## Alternatives considered

- **No ADRs, rely on specs.** Rejected — specs describe the target; ADRs explain *why* we chose that target. The v1 spec set exists and was still outrun by drift.
- **ADRs only in wiki / Notion / Confluence.** Rejected — detaches decisions from the code history, breaks offline review, and requires a separate permissions model.
- **ADRs inline in service README files.** Rejected — fragments the record and makes cross-cutting decisions invisible.
- **Use RFCs (longer, more ceremonious).** Rejected for now — overhead too high for a project at SENTINEL's current headcount. RFCs can be layered on top later if needed.
